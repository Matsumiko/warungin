# MODULE-MAP.md

Map of important modules, ownership boundaries, and first files to inspect.

Use this to help agents start in the right part of the codebase. Keep it factual and durable; task progress belongs in runtime sessions.

## How To Add A Module

```markdown
## Module Name

Responsible for:

- Main responsibility

Main files:

- path/to/file.ts

Related rules:

- Business, security, data, or UX rule that affects this module.

Common tasks:

- Bugfix or feature type agents often handle here.

Module pitfalls:

- Constraint that is easy to miss for this module.
```

## Modules

## App Routing And Shell

Responsible for:

- TanStack Start app bootstrapping, file-based routes, root shell, providers, and SSR error handling.

Main files:

- `vite.config.ts`
- `src/router.tsx`
- `src/routes/__root.tsx`
- `src/start.ts`
- `src/server.ts`
- `src/routeTree.gen.ts`

Related rules:

- Keep `tanstackStart({ server: { entry: "server" } })` so the custom `src/server.ts` SSR error wrapper remains active.
- Do not reintroduce builder-specific Vite config or browser event bridge reporting.

Common tasks:

- Add or adjust routes.
- Change root metadata/providers.
- Debug SSR and route-level errors.

Module pitfalls:

- `src/routeTree.gen.ts` is generated by TanStack Router tooling; avoid hand-editing unless the generated file is intentionally being refreshed.

## POS Catalog And Screens

Responsible for:

- Turso-backed catalog data and POS/backoffice workflows.

Main files:

- `src/lib/catalog.ts`
- `src/lib/catalog.types.ts`
- `src/lib/db/`
- `scripts/db-setup.ts`
- `src/routes/cashier*.tsx`
- `src/routes/_app*.tsx`
- `src/routes/customer-display.tsx`
- `src/routes/print*.tsx`

Related rules:

- Product/category catalog data is persisted in Turso through `src/lib/catalog.ts` and scoped to the authenticated tenant.
- Product SKU uniqueness is scoped to `(tenant_id, sku)`, not global `sku`; keep schema, setup migration, direct product mutations, and CSV import aligned with that rule.
- Backoffice business lists and reports must load through Turso-backed server functions rather than local fixture arrays.
- Role access helpers in `src/lib/auth.ts` define route access, role home redirects, standalone route access, and server-side role checks. Keep sidebar visibility and route guards aligned with those helpers.
- Checkout persistence is wired through `createTransaction`; it writes transactions/items, decrements product stock, applies tenant tax/service settings, and feeds sales/payment/profit reports from `transactions`.
- Product/category/supplier/customer/outlet/expense/app-user/promotion create flows are wired to Turso. Product edit, customer detail/edit/delete, supplier edit/delete, promotion edit/toggle/delete, category edit/delete, outlet edit/soft-delete, product variants, stock adjustment/history, inventory stock report, stock-transfer request records, tenant settings persistence and runtime tax/service application, import/export file downloads, product CSV import/upsert, stock CSV import, shift open/close, device pairing requests/logout, purchase order draft/status updates, purchase order item creation, and PO item receiving into product stock are also wired. Payment provider behavior and full stock-per-outlet ledgering are still not wired.

Common tasks:

- Add or migrate catalog-backed workflows.
- Adjust dashboard/report/cashier screens.
- Prepare future integration points for real APIs.
- Extend remaining placeholder/detail/edit flows and harden workflows such as payment provider behavior, role checks, full stock-per-outlet ledgering, CSV validation previews, and full purchase order lifecycle.

Module pitfalls:

- Do not import `src/lib/db/*.server.ts` directly from route files; use route-safe server functions such as `src/lib/catalog.ts`.
- `bun run db:setup` is schema-only; it does not insert placeholder catalog rows.
- If a role can see a route, verify its loader server functions are permitted for that role; otherwise the route will render the root error boundary after login/redirect.

## Auth And Tenant Ownership

Responsible for:

- Tenant registration, user login, session lookup, route guards, tenant ownership for business data, rate limiting, account lockout, PIN-based cashier login, password reset, and session cleanup.

Main files:

- `src/lib/auth.ts`
- `src/lib/rate-limit.ts`
- `src/lib/csrf.ts`
- `src/lib/backoffice.ts`
- `src/lib/catalog.ts`
- `src/lib/db/schema.ts`
- `scripts/db-setup.ts`
- `src/routes/login.tsx`
- `src/routes/register.tsx`
- `src/routes/forgot-password.tsx`
- `src/routes/reset-password.tsx`
- `src/routes/_app.tsx`
- `src/start.ts` (middleware chain)
- `src/server.ts` (security headers, CSRF cookie injection)

Related rules:

- New registration creates a tenant and owner user; login creates a Turso-backed session stored in an HttpOnly `warungin_session` cookie.
- Server functions must derive `tenant_id` from `requireAuthSession()` and include it in database predicates.
- Do not trust client-supplied tenant/user identifiers for reads or writes.
- Login rate limits: 5 attempts per IP per 15min, 10 attempts per email per 15min.
- Account lockout: 10 failed attempts per email triggers 15-minute lockout.
- Registration rate limit: 3 per IP per 5min. Password reset: 3 per IP per 5min.
- PIN-based login accepts 6-digit numeric PINs; PINs stored as scrypt hashes in `app_users.pin_hash`.
- Security headers are applied in `src/server.ts` on every response (X-Content-Type-Options, X-Frame-Options, Referrer-Policy, X-XSS-Protection, Permissions-Policy, HSTS in production).
- Session cleanup runs hourly, deleting expired sessions from the database.

Common tasks:

- Add authenticated server mutations for remaining POS/backoffice workflows.
- Extend logout/session invalidation behavior if multi-device session management is needed.
- Add role checks for manager/kasir/gudang/display permissions.
- Adjust rate limit thresholds or lockout rules.
- Integrate email delivery for password reset tokens.

Module pitfalls:

- `src/lib/auth.ts` should keep Node-only crypto imports inside server handlers so TanStack Start does not pull server-only modules into the client graph.
- Rate limiting is in-memory; multi-instance deployments need shared state (Redis) for production.
- CSRF protection uses a double-submit cookie pattern (`SameSite=Strict`) via `src/lib/csrf.ts`, validated in state-mutating server functions and injected in `src/server.ts`.
- MFA is not implemented.

## Notifications System

Responsible for:

- Real-time notification creation, persistence, filtering, mark-as-read, and unread count badge.

Main files:

- `src/lib/backoffice.ts` — `createNotification`, `getNotifications`, `markNotificationRead`, `markAllNotificationsRead`, `getUnreadNotificationCount`
- `src/routes/_app.notifications.tsx` — Notification list with filtering, mark-as-read, real-time SSE
- `src/hooks/useRealtimeEvents.ts` — `notification.created` event listener
- `src/components/app-sidebar.tsx` — Unread count badge on notification bell
- `src/routes/_app.tsx` — Unread count in layout loader
- `src/lib/db/schema.ts` — `notifications` table with `is_read`, `type` columns
- `src/lib/db/migrations/index.ts` — Migration 010 adds notification columns

Related rules:

- Notifications are gated by tenant setting toggles (e.g., `notifyLowStock`, `notifyCashDiff`).
- All notification queries are scoped by `tenant_id`.
- `createNotification` emits a `notification.created` SSE event for real-time delivery.

Common tasks:

- Add new notification types for additional events.
- Adjust notification settings toggles.

Module pitfalls:

- The `notifications` table previously had no `is_read` or `type` columns. Migration 010 adds them idempotently.
- Notification creation must be gated by tenant settings to respect user preferences.

## Barcode Scanner

Responsible for:

- Barcode scanner input detection via keyboard timing heuristic and product lookup.

Main files:

- `src/hooks/useBarcodeScanner.ts` — Detection hook (4-30 chars in <500ms = scanner input)
- `src/routes/cashier.index.tsx` — Integration with auto-add to cart, beep feedback, scan result overlay

Related rules:

- Scanner detection uses timing heuristic: characters arriving within 200ms gap + Enter within 500ms total.
- Manual typing never triggers scanner path due to >200ms inter-key gaps.
- Beep feedback uses Web Audio API (no external files needed).

Common tasks:

- Adjust scanner detection thresholds.
- Add barcode lookup for other pages (inventory, reports).

Module pitfalls:

- The "Scan" button focuses a hidden input; it does not open a camera or file picker.
- Barcode match is exact (`products.find(p => p.barcode === barcode)`), not substring.

## Low-stock Alerts & Quick Reorder

Responsible for:

- Low-stock product detection, sales velocity calculation, reorder suggestions, and PO auto-fill.

Main files:

- `src/lib/backoffice.ts` — `getLowStockProducts` (30-day velocity, suggested reorder qty)
- `src/routes/_app.dashboard.tsx` — Enhanced low-stock section with velocity data and estimated cost
- `src/routes/_app.inventory.index.tsx` — Low-stock alert banner with "Buat PO" quick action
- `src/routes/_app.purchase-orders.new.tsx` — "Isi dari Stok Rendah" Quick Reorder button

Related rules:

- `low.stock` events are emitted in `createTransaction`, `adjustProductStockHandler`, and `closeShiftHandler`.
- Sales velocity is calculated as `SUM(qty) / COUNT(DISTINCT DATE(created_at))` over 30 days.
- Suggested reorder = `avgDailySales * 7 + minStock - currentStock` (or `minStock * 2` if no sales data).
- Notifications are persisted alongside SSE events for low-stock conditions.

Common tasks:

- Adjust velocity calculation window or lead time.
- Add low-stock alerts to additional pages.

Module pitfalls:

- `getLowStockProducts` uses a subquery to find the most recent supplier from `purchase_order_items`. If no PO exists for a product, supplier is null.
- The dashboard uses `lowStockProducts.length` for the KPI count, not the old `products.filter(p.stock <= p.minStock)`.

## Testing

Responsible for:

- Unit and integration tests for server functions, auth helpers, rate-limiting, CSRF, and backoffice flows.

Main files:

- `vitest.config.ts`
- `src/__tests__/setup.ts`
- `src/__tests__/helpers.ts`
- `src/lib/rate-limit.test.ts`
- `src/lib/auth.test.ts`
- `src/lib/csrf.test.ts`
- `src/lib/auth.integration.test.ts`
- `src/lib/backoffice.integration.test.ts`

Related rules:

- `createServerFn` wrappers cannot be tested directly due to AsyncLocalStorage; test via extracted handler functions (`loginWithEmailHandler`, `logoutHandler`, `openShiftHandler`, `closeShiftHandler`).
- Mock DB via `createMockDb()` from `src/__tests__/helpers.ts`; configure per-SQL-pattern return values with `__setResult()`.
- Use `createMockSession(role)` for auth session fixtures.

Common tasks:

- Add tests for new server functions.
- Extend integration test coverage to transaction checkout and remaining handlers.

Module pitfalls:

- `vi.mock()` must be declared before imports due to Vitest hoisting.
- TanStack Start compiled packages (`@tanstack/start-client-core`, `@tanstack/start-storage-context`) resist standard `vi.mock()` interception.
- `loginWithEmailHandler` fetches IP internally via `getRequestIP()`; do not pass `ip` parameter.
